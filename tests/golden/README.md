# Golden test corpus

Pixelmaps and Karlin parameters pinned from C dotter
(`third_party/seqtools/dotterApp/dotter`).

Per spec §8.1 and CLAUDE.md, tolerance is **zero**. Any deliberate
algorithmic change must bump `PIXELMAP_FORMAT_VERSION` in
`crates/dottir-core/src/lib.rs` and ship an ADR.

## Layout

```
tests/golden/
├── karlin/        # JSON files pinning (lambda, k, h, window_size)
├── blastn/        # binary pixelmaps (Vec<u8>) + .sha256
├── blastp/
├── blastx/
└── self/
```

Each golden is paired with a `<name>.cmd` file recording the exact C dotter
invocation that produced it, so the corpus is regenerable:

```text
$ cat tests/golden/karlin/dna_chr4_self.cmd
dotter -W 0 examples/chr4_ref_seq.fasta examples/chr4_ref_seq.fasta
```

## Regenerating

1. Build C dotter from `third_party/seqtools/` (requires GTK2 dev headers
   plus glib-2.0). On Debian/Ubuntu:
   ```sh
   sudo apt install libgtk2.0-dev libglib2.0-dev autoconf automake libtool
   cd third_party/seqtools && ./configure && make
   ```
2. Run the corpus inputs against the resulting `dotter` binary, redirecting
   the Karlin-statistics stderr block into the corresponding `karlin/*.json`
   file (parser TBD; see `tests/regen.rs`).
3. Bump `PIXELMAP_FORMAT_VERSION` (only if the algorithmic contract
   changed) and add an ADR.

## Status

- ☑ `karlin/values.tsv` — pinned. Generated by `tests/golden_gen/karlin_ref`,
  a standalone C harness that extracts `karlin()` and `winsizeFromlambdak()`
  from `dotterKarlin.c` verbatim (no GTK dependency). The Rust port at
  `crates/dottir-core/src/karlin.rs` matches these values bit-for-bit on
  λ/K/H/window — verified by `crates/dottir-core/tests/karlin_golden.rs`.
  Regen: `tests/golden/karlin/regen.cmd`.
- ☐ `tests/regen.rs` — driver that calls the full C dotter binary and
  writes pixelmap goldens. Pending Phase 1 (currently we don't compute
  pixelmaps yet).
